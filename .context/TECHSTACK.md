# TECHSTACK

TokenTally is a cross-platform desktop application that reads Claude Code JSONL transcripts and renders a token-usage dashboard. The Go backend is wrapped in Wails v2; the embedded frontend is a Vue 3 SPA bundled by Vite. SQLite (pure-Go driver) is the persistence layer.

## Language and Runtime
- Go 1.25.7 (`go.mod`)
- Node.js 20 ‚Äî frontend build only, not a runtime dependency (`.github/workflows/release.yml`)
- TypeScript 5.4 ‚Äî compiled to JS at build time (`frontend/inspector/package.json`)
- WebView2 (Windows), WebKit (macOS), webkit2gtk-4.1 (Linux) ‚Äî provided by Wails

## Core Frameworks and Libraries
- Wails v2 v2.12.0 ‚Äî desktop shell, JS‚ÜîGo bindings, embedded asset server (`github.com/wailsapp/wails/v2`)
- getlantern/systray v1.2.2 ‚Äî Windows system tray icon and menu
- golang.org/x/sys v0.42.0 ‚Äî Windows SCM service handler (`golang.org/x/sys/windows/svc`)
- godbus/dbus/v5 v5.1.0 ‚Äî indirect, Linux desktop notification transport
- jackmordaunt/go-toast v2 ‚Äî indirect, Windows toast notifications
- wailsapp/go-webview2 v1.0.22 ‚Äî indirect, WebView2 loader

## Frontend Stack
- Vue 3.5 with `@vitejs/plugin-vue`
- Vue Router 4.3
- Pinia 2.2 (state)
- Vite 5.3 ‚Äî IIFE library build to `frontend/web/app.bundle.js` (`frontend/inspector/vite.config.ts`)
- DOMPurify 3.4 ‚Äî sanitization for HTML rendered from prompts
- marked 15 ‚Äî Markdown rendering
- gpt-tokenizer 3.4 ‚Äî client-side token counting
- A second, vanilla-JS SPA at `frontend/web/app.js` with a hash router (no build step) co-exists with the Vue bundle and is served by Wails as embedded assets

## Data and Persistence
- SQLite via `modernc.org/sqlite` v1.49.1 ‚Äî CGO-free pure-Go driver
- Database file: `~/.claude/tokentally.db` (overridable via `TOKENTALLY_DB`); WAL mode
- Connection pooling via `internal/db.Pool` ‚Äî separate read pool (`MaxOpenConns=4`) and single-writer pool
- Schema migrations tracked through a `schema_version` row in the `plan` table; migrations live in `internal/db/db.go`
- No ORM ‚Äî hand-written parameterised SQL throughout `internal/db`

## API and Contract Tooling
- No HTTP/gRPC/GraphQL surface. The only contract is the Wails JS‚ÜîGo binding generated under `frontend/web/wailsjs/` and called as `window.go.app.App.*()` from the SPA
- JSONL parsing of Claude Code transcripts at `~/.claude/projects/<slug>/<session>.jsonl` (`internal/scanner`)
- Embedded `pricing.json` (override via `TOKENTALLY_PRICING_JSON`) supplies model rates and plan metadata

## Security and Secrets
- No runtime secrets handled by the application; reads only local user files
- CI secret: `HOMEBREW_TAP_TOKEN` ‚Äî fine-grained PAT with `contents:write` on `payfacto/homebrew-tap`, stored in GitHub repo settings; consumed by the `brew-tap` job in `.github/workflows/release.yml`
- Windows install/uninstall paths require admin elevation through PowerShell `Start-Process -Verb RunAs` (`app/service_windows.go`)
- Frontend HTML insertion is gated by `fmt.htmlSafe()` and DOMPurify for user-derived strings

## Build and Dependency Management
- Go modules (`go.mod`, `go.sum`)
- npm with the lock file generated by `npm install --prefix inspector` (Wails runs lifecycle commands from the `frontend/` directory)
- Wails CLI orchestrates the combined Go + frontend build (`wails.json`)
- Version injection at build time via `-ldflags "-X 'tokentally/internal/version.Version=‚Ä¶'"` resolved from `git describe --tags --always --dirty`

## Testing Stack
- Go standard library `testing` ‚Äî no third-party test framework
- Test files distributed across packages: `internal/db`, `internal/scanner`, `internal/pricing`, `internal/tips`, `internal/classify`, `app`, `svc`, `integration`
- Tests run on any platform (use `:memory:` SQLite); some `_linux_test.go` files are platform-gated for environment-specific paths

## CI/CD and Delivery
- Bitbucket Pipelines (`bitbucket-pipelines.yml`) ‚Äî sole job force-pushes `main` and `v*` tags to GitHub mirror; image `alpine/git:latest`, `clone: depth: full`
- GitHub Actions (`.github/workflows/release.yml`):
  - Matrix build on `ubuntu-latest`, `windows-latest`, `macos-latest`
  - `actions/setup-go@v5` (Go 1.25), `actions/setup-node@v4` (Node 20)
  - Linux build needs `libgtk-3-dev`, `libwebkit2gtk-4.1-dev`, `libayatana-appindicator3-dev`, `xvfb` (used as `xvfb-run` because `wailsbindings` is a GTK app); compiled with `-tags webkit2_41`
  - Artifacts: `tokentally-linux-amd64.tar.gz`, `tokentally-windows-amd64.zip`, `tokentally-darwin-arm64.zip`
  - GitHub Release auto-published on `v*` tags via `gh release create/upload`
  - `brew-tap` job updates `Formula/tokentally.rb` (cask) in `payfacto/homebrew-tap` and pushes the bump commit
- No SAST, linter, or formatter steps in CI

## Infrastructure and Deployment
- No server-side infrastructure ‚Äî desktop application distributed as binaries
- Distribution channels:
  - GitHub Releases (Linux tarball, Windows zip, macOS arm64 zip)
  - Homebrew cask via `payfacto/homebrew-tap` (`brew install --cask payfacto/tap/tokentally`)
- Windows runtime modes selected by flag (`--service`, `--install`, `--uninstall`); SCM service registration plus a startup registry key handle background scanning

## Messaging and Async Processing
- In-process only: `app.scanLoop` runs a 30 s ticker that triggers the JSONL scanner and emits a Wails `"scan"` event consumed by the frontend (`window.runtime.EventsOn`)
- No external message broker

## Developer Experience Tooling
- Wails dev mode (`wails dev`) for live-reload macOS development
- `wails build ‚Ä¶ -skipbindings` for faster iterative Windows builds
- `frontend/web/wailsjs/` is gitignored and regenerated by `wails build`
- Project conventions, env vars, and architecture notes documented in `CLAUDE.md`
